For the chorus-review project (`/home/az/code/chorus-review`): roster size is a user-experience budget paid by whoever runs a round (RSVP dispatch per persona per round, Phase 0.5 in `skill/chorus-review/SKILL.md`). Adding a persona to fill an empty cell on a coverage chart is roster-growth-serving-itself, not user benefit.

**Why:** The 002 ninth persona (constraint-and-flow-advisor) is *justified* — it traces to a real user-stated goal ("i feel like the chorus is disbalanced", spec.md:9) and the abstain path proves it was built for the user, not the roster. But the coverage chart's "Reading of the whole" (`docs/reviews/2026-06-05-chorus-coverage-map.html:228-232`) argues for a TENTH (systemic-quality) persona purely because performance/data-integrity axes score below level-3. An empty chart cell is the roster noticing an asymmetry in itself — it is NOT evidence a user's goal went unmet. Coverage maximisation is excise the moment it stops changing the user's decision; more voices = longer round + diluted signal, with the one decision-changing finding buried.

**How to apply:** When any future chorus feature proposes adding a persona, demand the user-need evidence: did someone run a real round, hit a wall, and find no voice there? If the only justification is "axis X has no champion" / "fill the chart" / "rebalance for completeness," name it as feature-as-goal and propose seating latent expertise (e.g. the unseated bjarne-stroustrup-persona) per-round on demand instead of standing up a permanent voice. The contract for a new seat is an observed unmet need, never an empty cell. Note the chart already applies this discipline against "a second product voice" — hold it to the same standard in every direction.
